# API Conventions

## Base Path and Versioning

Pattern: `/api/v{n}/{resource}`

Examples: `/api/v1/users`, `/api/v1/auth`

Services registered in `configfiles/gateway.yml` exposed via gateway under `/api/<service>/...`.

## HTTP Method Semantics

| Method | Use |
|---|---|
| GET | Fetch by ID or list with filters (no body) |
| POST | Create new resource or complex query (body required) |
| PUT | Full replace of resource (all fields required) |
| PATCH | Partial update (map of changed fields) |
| DELETE | Soft or hard delete by ID |

Abstract controller provides these automatically — see `AbstractJOOQDataController` for default implementations.

## Response Wrapping

Return `ResponseEntity<T>` everywhere. No custom envelope.

```java
return ResponseEntity.ok(result);
return ResponseEntity.created(uri).body(created);
return ResponseEntity.noContent().build();
```

Error responses: RFC 7807 `ProblemDetail` — auto-generated by `ControllerAdvice`.

## Pagination

Use Spring `Pageable` from `spring-data-commons`.

Standard query params (Spring auto-binds):
- `page` — zero-indexed, default 0
- `size` — default 20
- `sort` — field,direction (e.g. `sort=createdAt,desc`)

Return type: `Page<DTO>` or `Slice<DTO>`.

Abstract controller `readPageFilter(Pageable, MultiValueMap<String,String>)` also accepts arbitrary filter params as query string.

## Request Validation

- Put `@Valid` on `@RequestBody` params
- Jakarta validation annotations on DTO fields: `@NotNull`, `@NotBlank`, `@Size`, `@Email`, `@Pattern`
- No `@NoSpecialCharacters` yet — use `@Pattern` for character constraints
- Validation errors surface as 400 via `ControllerAdvice`

## Logging

Log at method entry with correlation ID from MDC (`x-debug` key in log pattern). Log: operation name, entity ID. Do NOT log: passwords, tokens, full DTO content (PII risk).

Pattern set in `configfiles/application.yml` logback config.

## @ApiVisibility — Tier Assignment

Class-level annotation sets default tier for all methods; method annotation overrides.

```java
@ApiVisibility(Visibility.INTERNAL)   // class default
public class MyController {
    @ApiVisibility(Visibility.EXTERNAL)  // override for this method only
    public ResponseEntity<Foo> publicEndpoint() { ... }
}
```

Only put `@ApiVisibility` on methods that differ from class default.

## Tag Placement

- `@Tag(name = OpenApiConfig.TAG_*)` on controller class
- Never on methods
- Never in `@Operation(tags = ...)`

## @Operation Format

```java
@Operation(
    summary = "Short action phrase under 60 chars",
    description = "One or two sentences explaining context, side effects, or non-obvious behaviour."
)
```

No `tags`, no `operationId` override, no `responses` block unless adding non-standard status codes.

## DTO Naming

| Type | Naming |
|---|---|
| Request/input | `CreateUserRequest`, `UpdateUserRequest` |
| Response/output | `UserDTO`, `UserResponse` |
| Internal transfer | No suffix, or `User` (plain domain name) |

DTOs in `<module>/src/main/java/studio/ikara/<service>/dto/`.

## Error Handling

- Service layer: throw `GenericException(HttpStatus, message)`
- Controller layer: no try/catch — let `ControllerAdvice` handle
- External boundary (e.g. Feign, HTTP client): catch checked exceptions, wrap in `GenericException`
- Never expose stack traces or internal exception messages to API consumers

## New Endpoint Checklist

- [ ] Controller extends appropriate abstract controller or standalone `@RestController`
- [ ] `@Tag(name = OpenApiConfig.TAG_*)` on controller class
- [ ] `@ApiVisibility` on class (INTERNAL default); method override if EXTERNAL/BACKOFFICE
- [ ] `@Operation(summary, description)` on method
- [ ] Request body has `@Valid`; DTO fields have validation annotations
- [ ] No PII in log statements
- [ ] Path follows `/api/v{n}/{resource}` convention
- [ ] Added to permitted paths in security config if public endpoint
